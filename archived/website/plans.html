New Website Options

There's been a couple of discussions going on about what's the best route to go down for redesigning the theatre website. As of now two main options have been proposed:

Self-Hosted

We would host a website, built awesomely by Will, ourselves.

Pros

  • Maximum flexibility
    • We choose the CMS (content management system, the software that we use to build the site). Will's advice is for WordPress (see http://www.nikkipilkington.com/why-wordpress-is-a-good-choice-for-your-website/)
    • http://blogs.nottingham.ac.uk is a WordPress site, the university obviously see the value in using the tool if they use it themselves.
    • Website can be updated from anywhere, will be most user-friendly backend.
    • Site will work how we work (if that makes sense)
  • We can build our own booking system that works exactly as how we want it to. Should still be able to use uni's payment gateway.
  • Design
    • Professional! Huge improvement on the current site.
    • Much more user friendly (for visitors).
    • Built with strong knowledge of the theatre.
    • Lots of images! Making the most of all of the very high quality images we have of past and present shows.
  • Ability to make rapid changes to design.
  • Very reliable set-up & site/server optimisation therefore faster load times → better position in Google results

Cons

  • Cost to rent a server (£5/month)
  • We train ourselves
  • Reliance on Will (he is very good though :) / a technical person in the theatre

University Hosted

The University Web team would host a site built by them with lots of input from us OR a site built by Will (may/may not be feasible).

Pros

  • Ease of use
    • Still user friendly! CMS used by the uni is designed for people who don't necessarily have any experience with managing websites so it is very easy to manage and can be updated from anywhere.
  • Support
    • The comms team are very happy to help, and are willing to support the site throughout it's life.
    • Consistent source of training - official sessions held up to 15 times a year so constant opportunities for committee members to learn should they want to as well as the option of a private session at the start of the year.
    • Option of using (or at least trialling) the University's payment system if we did want to offer ticket purchasing on the site.
  • Design
    • Professional! Huge improvement on the current site.
    • Much more user friendly (for visitors).
    • Team based at Kings Meadow with lots of experience who would be happy to enact any major changes.
    • Lots of images! Making the most of all of the very high quality images we have of past and present shows.

Cons

  • Limitations
    • Uni CMS is designed to be used corporately rather than for a student theatre website - will mean working hard to get the site having the right feel and not being too 'clunkly', boring, etc
    • Big changes to design would need to be enacted by the Web team (therefore not immediate!)
    • Issues/problems (e.g. with last-minute uploads) could be slow to fix.
  • Lack of control, university control the site.

What Needs To Be Done

  • Decide on course of action
  • Regain control of newtheatre.org.uk domain name (& give the wiki a better address)

Lifetime of project

Considering the life of the current site (3 years) and the pace at which web design moves, the new site is expected to need to last 4-5 years.

After every 6 months (potentially at the end of each season?) of the new site being in service, a review should take place - using both user and visitor stats - to find what works and what doesn't. Additional changes can then be made. Depending on the results of this, the process may need to be repeated. If successful this 'evolutionary' approach will increase the lifetime of the new website.
